JanusFlow: Harmonizing Autoregression and Rectified Flow for Unified Multimodal Understanding and Generation

논문 링크

한 줄 요약 (TL;DR)

JanusFlow는 Rectified Flow + 분리형 비전 인코더 + 표현 정렬로 1.3 B 파라미터 단일 모델만으로 이미지 이해·생성 양쪽에서 동급 최고 성능(FID 9.51 / GenEval 0.63 / MMBench 74.9)을 달성한다.

핵심 아이디어

“LLM 내부에 연속형(ODE) 생성기를 내장하고, 이해·생성에 각기 최적화된 인코더를 연결한 뒤 중간 표현을 정렬하면, 작은 모델도 멀티모달 SOTA를 동시에 이룰 수 있다.”

  • Rectified Flow → 30 스텝으로 고품질 샘플링 (Diffusion 대비 ≈6–8 × 빠름)
  • SigLIP-Large(이해) + ConvNeXt VAE(생성) 분리 → 작업 간 간섭 제거
  • REPA 정렬 → 시맨틱·시각 품질 동시 향상 (FID –2.2 / POPE +6.6 pt)

배경: 그들이 해결한 문제

멀티모달 LLM 분야는

  1. LLM + 외부 Diffusion: 고품질이지만 두 모델·긴 샘플링 루프 → 지연·메모리 ↑.
  2. VQ-AR 통합: 토큰 양자화 한계로 세밀도 ↓.
  3. 공유 인코더 통합: 이해·생성 특성 충돌 → 양쪽 성능 모두 감소. JanusFlow는 단일 모델로 이 세 문제(복잡도·세밀도·간섭)를 모두 해소하려 한다.

새로운 접근법: JanusFlow

작동 원리: 구체적인 예시로 살펴보기

  1. 텍스트 입력: “draw a small black frame”

  2. 생성 단계

    1. 노이즈 $z_0∼\mathcal N(0,I)$ (48 × 48 × 4).
    2. LLM → velocity $v_\theta(z_t,t|x_\text{con})$ 예측.
    3. ODE 적분 $z_{t+Δt}=z_t+v·Δt$ (Δt≈1/30) × 30 스텝.
    4. ConvNeXt decoder가 $z_1$ → 3 × 384² RGB.

  3. 이해 단계

    1. 384² 이미지를 SigLIP로 576 토큰 추출.
    2. 텍스트·이미지 토큰을 한 시퀀스로 LLM에 넣어 “It is a black frame with a white center.” 응답. → 하나의 그래프가 두 과제를 모두 소화한다.

우리의 관점: 강점, 한계, 그리고 이 연구가 중요한 이유

강점

  1. 효율적 통합 – 두 개 대형 네트워크를 하나로 축소, 추론·메모리 비용 절감.
  2. Scale-efficient SOTA – 1 B 대 모델로도 대형(7 B+) 모델 성능에 근접하거나 추월.
  3. 단순 설계 – 패치 토큰 concat 만으로 텍스트·이미지 정보를 자연스럽게 교차.

한계

  1. 카운팅·도표 추론 약점 – GenEval Counting, ChartQA 지표에서 경쟁 모델 열세.
  2. 해상도 확장 – 256²로 낮추면 이해 성능 2 – 3 pt 하락, 4K 이상 실험 미제공.
  3. 학습 비용 – 총 1,600 A100 GPU-days (저자 보고) → 에너지·탄소 부담.

왜 중요한가?

  • 모바일·RAM 16 GB 한계 기기에서도 실용적 멀티모달 AI 구현 가능성.
  • 통합 설계 패러다임 전환: “LLM + ODE” 조합이 Diffusion 대안을 제시.

다음 단계는?: 앞으로의 길

  1. 고해상도 (768² ↑) Rectified Flow 연구로 FID 추가 하락 및 포토 리얼리즘 확보.
  2. 수량·도표 인식 특화 데이터 추가해 Counting·ChartQA 취약점 개선.
  3. 비디오/3D 확장: ODE 샘플러를 시공간-연속 생성기로 일반화.
  4. 경량화·양자화: SigLIP 300 M → 모바일 ViT 대체, 실제 온-디바이스 배포 검증.

Bottom line — JanusFlow는 “작지만 다재다능한 멀티모달 LLM”의 실현 가능성을 증명했으며, 앞으로의 과제는 범용성 확대와 효율 최적화다.

TL;DR

JanusFlow 는 1.3 B 파라미터의 단일 LLM 안에 자연스러운 이미지 이해와 고품질 이미지 생성을 모두 담으면서, 기존 통합 모델의 구조적 복잡성과 성능 저하를 동시에 해소한다. 핵심 아이디어는 Rectified Flow (연속 잠재 공간 생성)와 LLM 자동회귀를 디코플드(분리형) 비전 인코더로 연결하고, **중간 표현 정렬(Representation Alignment)**으로 두 작업의 시맨틱 간극을 줄인 것이다. 이로써 384×384 해상도에서 FID 9.51·GenEval 0.63·MMBench 74.9를 달성, 동급(≈1 B)·전용 모델을 모두 앞선다.

1. 핵심 연구 공백 (Research Gap)

번호공백 내용왜 중요한가?
①단일 모델 안에서 이해·생성을 모두 다루는 간결한 프레임워크 부재기존 방법은 LLM+Diffusion 이중 구조 → 파이프라인 복잡, 추론 비용↑
②공유(Shared) 이미지 인코더 사용 시 ‘작업 간 간섭’ 문제같은 특성을 이해·생성에 동시에 쓰면 성능이 양쪽 모두 감소
③벡터 양자화(VQ) 기반 통합 모델의 토큰화 한계이미지 토큰 품질↓ → 세밀한 시각 정보 손실
④Flow/Diffusion 계열을 LLM 문맥 안에 자연스럽게 삽입하는 방법 미확립서로 다른 학습 목표(확률 흐름 vs. 언어 예측)를 동시에 최적화하기 힘듦

요약: JanusFlow는 위 4가지 공백을 Rectified Flow + 분리형 인코더 + 표현 정렬이라는 조합으로 해소한다.

관찰: 동일 규모(≈1 B) 기준으로 JanusFlow가 *시각 품질(FID)*과 텍스트-이미지 정합(GenEval) 모두 기존 최고치를 경신하며, 이해 성능도 5 ~ 11 pt 상승. 이는 Rectified Flow + 표현 정렬의 효과를 수치로 검증한다.

참고 메모

  • Rectified Flow: ODE 기반 한 단계(continuous) 샘플링 → Diffusion 대비 단계 수 ↓, LLM 안에 벡터 필드 예측으로 자연스럽게 삽입.
  • Representation Alignment: 𝑓_enc (SigLIP) 특징과 LLM 중간 벡터를 코사인 정렬 → FID ≈ −1.2, CLIP +1.5 ↑ (256해상도 실험).
  • Decoupled Encoders: 이해용 SigLIP-Large vs. 생성용 ConvNeXt 쌍 → 작업 간 간섭 방지, 각각 최적 특성 공간 확보.

이로써 JanusFlow는 *“단일 1 B급 모델로 SOTA 이해·생성을 동시에”*라는 미해결 질문에 실질적 답을 제시한다.

한줄 결론

JanusFlow는 Rectified Flow를 LLM에 직결하고 이해·생성용 인코더를 분리한 뒤 표현 정렬(REPA)을 얹어, 1.3 B 파라미터로 FID 9.51 / GenEval 0.63 / MMBench 74.9를 달성한 최초의 통합 모델이다.

요약 해설 (저자 입장)

  1. “단일 네트워크로 충분하다” – Rectified Flow를 LLM에 그대로 끼워 넣어 두 개의 거대한 모델(LLM + Diffusion)과 긴 샘플링 루프를 없앴다. 그래서 동일 해상도 기준 추론 속도·메모리를 모두 절감하면서도 FID가 더 낮다.
  2. “분업이 답이다” – 이해와 생성은 필요 특성이 다르므로 동일 인코더를 공유하면 서로를 방해한다. SigLIP vs ConvNeXt 분리는 이 간섭을 제거해 양쪽 점수를 동시에 끌어올렸다.
  3. “보이지 않는 중간을 맞춰라” – REPA로 두 인코더의 의미 공간을 접목시켜 Flow 샘플이 언어 조건을 더 정확히 반영하게 만들었다.

결국 저자들은 *“작지만 똑똑한 1 B급 올인원 모델”*이라는 새 설계 공간을 제시하며, 실험적으로도 전용·대형 모델을 능가함을 보여 준다.

TL;DR

JanusFlow는 **“이해 = LLM 자동회귀 + 생성 = Rectified Flow”**를 한 몸에 담는다.

  1. 분리형 인코더(SigLIP·ConvNeXt)로 작업 간 간섭을 제거하고
  2. Rectified Flow ODE를 LLM 토큰 루프에 삽입해 30 스텝만에 이미지를 뽑아내며
  3. REPA 정렬로 두 인코더 표현을 일치시켜 FID 9.51 / GenEval 0.63 / MMBench 74.9를 달성한다.

Rectified Flow ODE

$$ \frac{d\mathbf z_t}{dt}=v_\theta(\mathbf z_t,t),\qquad \mathbf z_0\sim\mathcal N(\mathbf 0,\mathbf I)\tag{1} $$

— LLM이 velocity field $v_\theta$ 를 직접 예측해 노이즈 $\mathbf z_0$ 를 데이터 분포로 곧장 운반하는 1-차 미분 방정식이다.

1. 어떤 역할을 하나?

기능왜 중요한가?수치 효과
Continuous transport(1)만 적분하면 단 한 네트워크(LLM)로 텍스트→이미지 경로가 완성된다.30 ODE 스텝으로 384² 이미지를 생성 → 기존 Diffusion(≤250 스텝) 대비 최소 8× 빠름.
LLM-내장 가능velocity 는 토큰-조건을 입력 받아 LLM이 바로 계산 ⇒ 별도 Diffusion UNet 불필요, 파이프라인·메모리↓.1.3 B 파라미터 ‘단일’ 모델 구성이 가능.
고품질 점진적 복원선형 경로 $t\mathbf x+(1-t)\mathbf z_0$ 과의 유클리드 오차로 훈련 → long-range 이동도 안정.FID 9.51 (↓36 % vs. Show-o 15.18)·GenEval 0.63 SOTA.

2. 왜 JanusFlow의 ‘성공 열쇠’인가?

  1. 구조 단순화 – (1)을 LLM 토큰 루프에 삽입하면 언어 이해와 이미지 생성이 동일 그래프에서 끝난다. 따라서 분리형 인코더·표현 정렬 같은 추가 기법이 효과적으로 결합될 수 있는 토대가 마련된다.
  2. 추론 속도·메모리 동시 절감 – 추론은 velocity → ODE 한 걸음의 반복이므로, 디퓨전 대비 매번 거대 UNet forward 가 사라진다. 이는 모바일·RAM 16 GB 한계 환경에서도 실제 deploy 를 가능케 한다.
  3. 해상도 확장성 – ODE는 스텝 수가 해상도와 무관해 고해상도일수록 상대 효율이 더 높아진다.

요약: 식 (1) 하나가 “LLM 내부에 Flow를 접목” 하는 설계의 열쇠이며, 이 덕분에 JanusFlow는 ‘작지만 올인원’ 모델로 이해 + 생성 모두 최첨단이라는 결과를 구현한다.

MMMU·MM-Vet 같은 고차 추론(29 ~ 31 점)일부 전문 벤치마크는 LLaVA-v1.5 등 7 B 모델에 뒤짐저자는 “학습 데이터에 해당 과목 도메인 강의가 부족” 탓으로 분석하고, future work 로 도메인 증설을 제시
ChartQA 64.6 – Qwen-VL-Chat 66.3보다 낮음도표·표 인식에서 완전 우위 미달인코더 분리가 시각 패치 처리엔 유리하지만, 구조적 문서 도표에 특화된 모듈이 없다는 점을 한계로 언급

4. 요약 인사이트

  • Scale-efficient SOTA: 1.3 B로 이해·생성 쌍방 계열 최고치를 동시에 경신.
  • 대형 모델과 격차: 생성 품질은 7 B급 최고와 1.8 FID 차이, 이해 대부분 지표는 오히려 앞섬 → 모바일·16 GB RAM 배포 관점에서 매력.
  • 개선 포인트: (i) 도표·멀티-도메인 데이터 확충, (ii) 하이-레졸 모델(>512²)로 FID 추가 하락, (iii) instruction 세밀도 강화를 위한 인-컨텍스트 학습.

결론적으로, JanusFlow는 *“소형 단일 모델로 두 생태계 SOTA 달성”*이라는 목표를 대체로 완수했으며, 남은 공백은 규모·데이터 특화 확장으로 해결 가능하다는 것이 저자들의 입장이다.

3. 핵심 인사이트

  1. ‘드러난 한계가 적다’는 것이 오히려 한계

💡 왜 이 증강 조합이 효과적인가?

  1. 해상도 통일 = 토큰 정규화

    • LLM 입력에서 이미지 패치는 고정 토큰 길이로 투영됨. 해상도를 맞추면 패치 시퀀스 길이 불확실성이 사라져 positional embedding 낭비를 줄인다.

  2. 패딩 vs. 크롭, 과제별 최적화

    • 이해: 질문-응답·객체 탐지에서는 전체 시각 맥락이 중요 ⇒ 패딩이 안전.
    • 생성: Flow 샘플링은 경계 픽셀까지 노이즈를 풀어야 하므로, 패딩이 있으면 경계 ≠ 콘텐츠로 학습돼 FID 악화. 랜덤 크롭은 이를 방지하고 데이터 다양성까지 증가시킨다.

  3. 텍스트 드랍-아웃과의 시너지

    • 생성 단계에서 프롬프트가 없을 수도 있다는 ‘노이즈 조건’을 학습시켜 CFG(= w=2) 추론 시 모드 충돌을 줄이고, FID를 9.51→ 9.4까지 낮추는 데 기여했다 (sampling-step ablation).

한줄 결론

JanusFlow는 *이해(vision–language)*에는 SigLIP-Large-Patch/16 ViT 백본을, *생성(rectified-flow)*에는 ConvNeXt 기반 VAE encoder/decoder를 분리 적용한다. 이미지 패치는 16 × 16 공간격자를 토큰으로 변환해 LLM 임베딩 시퀀스에 직접 삽입하고, 생성 쪽은 **SDXL-VAE latent (48 × 48)**를 ConvNeXt로 임베딩해 LLM과 연동한다. 탐지·분할 전용 모듈(RPN·마스크 헤드)은 없고, 객체 위치 질의는 언어 토큰으로만 출력한다.

파이프라인백본 & 해상도공간 특징 추출LLM 융합 방식
멀티모달 이해SigLIP-Large-P/16 (ViT)
입력 384² → 24×24×Denc 피처		- 16×16 패치 + CLS
- 24×24 patch tokens → flatten
- Linear proj→Demb		[텍스트 토큰] + |BOI| + 576 image tokens + |EOI| 시퀀스에 concatenate 후 Transformer self-attention
텍스트→이미지 생성ConvNeXt blocks (encoder 𝑔enc, decoder 𝑔dec) + SDXL-VAE latent 48×48×4- 시작 노이즈 𝑧₀ (48×48×4) → ConvNeXt downsample → 48×48 tokens
- Time-step embedding 1 token 추가		[프롬프트 토큰] + |BOI| + 2304 latent tokens + t-token 입력 → LLM이 velocity field 예측, 30-step ODE 후 𝑔dec 재구성

융합 설계 요점

  1. **Linear projection → 동일 Demb**로 맞춰 텍스트·비전 토큰이 동일 어텐션 공간에서 교차 정보 공유.
  2. SigLIP와 ConvNeXt를 완전 분리해 ‘task interference’ 제거; Ablation에서 POPE +6.6 pt·FID –2.2 개선을 확인했다.
  3. 객체 탐지·세그멘테이션 전용 헤드는 없으며, POPE처럼 위치가 필요한 질문은 “The cat is at top-left” 같은 언어 답변으로 처리—bounding-box 회귀나 anchor box는 사용되지 않는다.

정리: JanusFlow의 핵심은 SigLIP-ViT로 고수준 시맨틱을, ConvNeXt VAE로 저수준 공간 복원을 담당하게 하여, 동일 Transformer 시퀀스 안에서 두 종류 토큰을 단순 연결만으로 효과적으로 융합한 점이다.

▸ 실패 — 다중 객체 개수 추론 오차

GenEval Counting 점수가 0.45 로 하위 항목 중 최저. 실제 Appendix 예시에서 펭귄 수를 11 → 9 로 잘못 응답하거나, 막대그래프 카운트를 빼먹는 사례가 보고됐다(문장 코멘트, Fig. 5). 이는

  2. 언어 토큰만으로 수량을 표현하는 디자인이 누적 오차를 유발한 것으로 해석된다.
